Astrolabe is a Swift package for declaring the desired state of a macOS machine.
You describe packages, launchd jobs, system settings, Jamf settings, and custom
steps in a Swift `body`; Astrolabe runs as a root LaunchDaemon and continuously
reconciles the real machine back to that declaration.

The framework deliberately borrows from SwiftUI:

- `Astrolabe` is the app entry point.
- `Setup` is the declaration protocol.
- `@SetupBuilder` turns multiple declarations into a tree.
- `@State`, `@Storage`, and `@Environment` drive reevaluation.
- Modifiers attach behavior and metadata to declarations.

This guide is written for developers who want to build on Astrolabe, debug an
Astrolabe setup, or extend the framework itself. It explains the public API and
the design reasons behind it, then points to the source files where those ideas
live.

## What Astrolabe is good at

Astrolabe is a good fit when you want a Mac to remain in a known state over
time:

- Install Homebrew packages and casks.
- Install `.pkg` artifacts from a catalog, GitHub releases, or a custom source.
- Write and optionally activate LaunchDaemons and LaunchAgents.
- Apply system settings such as hostname and power management.
- Apply Jamf settings such as computer name.
- Persist local setup decisions with `@Storage`.
- Attach user prompts, lifecycle hooks, and background tasks to declarations.
- Ship a setup binary that can update itself from GitHub releases.

Astrolabe is not a one-shot script runner. A script says "run these commands in
this order." Astrolabe says "this state should exist" and keeps checking whether
that remains true.

## Project shape

The repository is a SwiftPM package:

```text
Package.swift
Sources/
  Astrolabe/
  AstrolabeUtils/
Examples/
Tests/
docs/
```

The package exposes two libraries:

| Product | Purpose |
| --- | --- |
| `Astrolabe` | The main framework for declarative setup apps. |
| `AstrolabeUtils` | A lightweight storage client for other processes that need to read or write Astrolabe `@Storage` values. |

Important source landmarks:

| Area | Source files |
| --- | --- |
| Entry point | `Sources/Astrolabe/Astrolabe.swift` |
| Declaration model | `Sources/Astrolabe/Setup.swift`, `Sources/Astrolabe/SetupBuilder.swift`, `Sources/Astrolabe/SetupTypes/` |
| Engine | `Sources/Astrolabe/Engine/LifecycleEngine.swift`, `Reconciler.swift`, `TaskQueue.swift`, `LoopSupervisor.swift` |
| Tree and identity | `Sources/Astrolabe/Tree/TreeBuilder.swift`, `TreeNode.swift`, `NodeIdentity.swift` |
| State | `Sources/Astrolabe/State/`, `Sources/Astrolabe/Environment/` |
| Built-in declarations | `Sources/Astrolabe/Steps/` |
| Modifiers | `Sources/Astrolabe/Modifiers/` |
| CLI and daemon management | `Sources/Astrolabe/CLI/` |
| Self-update | `Sources/Astrolabe/Update/` |
| Telemetry | `Sources/Astrolabe/Telemetry/` |

## Platform requirements

Astrolabe is macOS-only:

- Swift 6.2 or newer.
- macOS 15 or newer, as declared in `Package.swift`.
- Root privileges at runtime.
- launchd, `installer`, `pkgutil`, `softwareupdate`, `scutil`, and Homebrew
  integration on the target Mac.

The Cursor Cloud/Linux environment can resolve Swift package dependencies, but
it cannot compile or run the Astrolabe targets because the source imports
macOS-only modules such as `Darwin` and `SystemConfiguration`. Build, test, and
run the package on a macOS 15 or newer host.

## Design anchors

The project's invariants live in
[`CONSTITUTION.md`](https://github.com/photon-hq/Astrolabe/blob/main/CONSTITUTION.md)
at the repository root. Two of the decisions it codifies are expanded in this guide:

- [Core Concepts](./core-concepts) explains why explicit retry APIs were removed —
  the reconcile loop is the only retry.
- [State and Persistence](./state-and-persistence) explains the shared storage
  coordinator behind `@Storage`.

The rest of this guide expands those ideas into practical developer guidance.
